## 一、项目起源

用户是一名 AI 产品经理，痛点：**竞品调研重复、格式化文档耗时**。想要一个工具——浏览竞品网页时一句话生成完整分析报告。

## 二、产品形态选择

讨论了四种形态：

| 方案 | 优点 | 缺点 |
|------|------|------|
| Web 应用 | UI 丰富，可独立部署 | 需要切换到浏览器外 |
| VS Code 扩展 | 编辑器内使用 | 产经不一定用 VSCode |
| CLI 命令行 | 终端交互 | 门槛高 |
| **浏览器插件** ✅ | 边浏览边生成，不切工具 | 需 Chrome Web Store 审核 |

**最终选择：浏览器插件（全兼容 Chromium 系：Chrome + Edge + Arc）**

交互方式：
- 侧边栏 (Side Panel) — 主力交互区，对话+报告预览
- 右键菜单 — 快速入口，选中关键信息一键提取到侧边栏

### 市场空白（我们的切入点）

| 缺失 | 我们的解决方案 |
|------|---------------|
| 无独立浏览器插件的 PRD 生成 | WXT 扩展，浏览任意网站即可生成 |
| 无 AI 产品经理专属模板 | 14 模块含 AI 产品专属维度（模型策略/Prompt 设计/数据策略/AI 风险评估） |
| 端到端一站式不存在 | 浏览竞品→分析→生成 PRD→导出，一个扩展全搞定 |
| 产经友好的交互 | 一键生成 + 对话微调，不写代码 |

## 五、PRD 报告 14 模块设计

经过讨论确定，一份完整 PRD 涵盖：

1. 产品背景 — 市场趋势、用户痛点、业务价值
2. 产品目标 — 可量化成功指标(OKR)
3. 竞品分析 — 功能矩阵、优劣势、差异化机会
4. 目标用户 — 用户画像、使用场景、核心需求
5. 用户故事 — "作为 XX，我希望 XX，以便 XX"
6. 功能需求 — 核心功能、优先级(P0/P1/P2)
7. 非功能需求 — 响应延迟、吞吐量、推理成本、公平性与伦理
8. 异常处理 — 每种异常的具体策略、提示文案、恢复路径
9. 边界条件 — 输入长度、文件格式、语言范围、并发上限
10. AI 可行性分析 — 输入弹性、规则可语言化水平、示例可得性、输出弹性、容错空间
11. 数据需求 — 数据来源、规模、标注规则、清洗规则、合规(GDPR/个人信息保护法)
12. 产品流程图 — Mermaid 格式
13. 数据埋点 — 追踪指标和埋点位置
14. 发布计划 — 版本节奏、里程碑、灰度策略

AI 产品额外维度：模型策略、Prompt 设计、数据策略、AI 风险评估

## 六、技术方案决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 扩展框架 | WXT（非 Plasmo/CRXJS） | Vite 驱动、HMR<100ms、Chrome/Edge/Firefox 全兼容 |
| 输出格式 | Markdown 优先 + Word/PDF 可选 | Markdown 兼容语雀/Notion/飞书/Obsidian；Word 适应传统企业 |
| 服务端 | 纯前端，零后端 | 用户自带 API Key，成本自付，隐私无隐患 |
| AI 模型 | 多模型，用户自选 | Claude 默认→可切 DeepSeek/OpenAI/豆包/Kimi |
| 状态管理 | Zustand（非 Redux） | <1KB，适合插件场景 |
| 页面抓取 | Readability.js（Mozilla） | 正文提取业界标准 |

## 七、输出格式演进过程

- 最初讨论输出 Markdown，用户问"Markdown 能转成 Word 吗"→ 确认可以（Pandoc/Typora/语雀都能转）
- MVP 只实现了 Markdown 下载 + 复制
- 用户实际使用后提出需求 → 升级为 Markdown + Word(.docx) + PDF(打印另存) 三格式

## 八、开发过程

采用 Subagent-Driven Development 模式，12 个 Task 逐任务实现+审查：

| 阶段 | 任务数 | 模式 |
|------|--------|------|
| 项目脚手架 | Task 1 | WXT + React + TS + Tailwind 初始化 |
| 数据层 | Task 2 | 共享类型 + Prompt 模板 + chrome.storage 封装 |
| 核心引擎 | Tasks 3-4 | Background SW(消息路由+Claude API) + Content Script(Readability 抓取) |
| UI 层 | Tasks 5-9 | Zustand Stores + App Shell + Chat UI + Settings + Export |
| 集成 | Task 10 | E2E 全链路打通：侧边栏→抓取→API→渲染→导出 |
| 质量 | Tasks 11-12 | 错误处理/重试/超时/Toast + README + 最终审查 |
| 修复 | 审查后 | XSS 防护(DOMPurify) + 多模型支持 + 对话历史 + Word/PDF 导出 |

17 次提交，23 个源文件，最终产物体积 775KB。

## 九、关键洞见

1. **输出格式不用死磕**：Markdown 是全宇宙最灵活的格式，其他格式都可以从它转
2. **浏览器插件 > Web 应用**：对标用户做竞品调研时就在浏览器里，不切工具最重要
3. **MVP 只做一件事**：浏览→一键生成→导出，Scout 的教训"scope creep 是最大敌人"我们记住了
4. **AI 产品经理需要专属模板**：通用 PRD 模板不够，模型策略/Prompt 设计/AI 风险评估是 AI PM 独有的需求
5. **Obsidian 集成不需要专门开发**：生成的 Markdown 自带 Frontmatter，直接拖进 Vault 就能用，v0.4 砍掉
